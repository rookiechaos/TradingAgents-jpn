# TradingAgents-jpn 日本客户化改造计划

## 1. 目标定位

将当前 `TradingAgents` 从通用型多智能体交易研究框架，改造成更适合日本客户使用的：

`日本株 AI リサーチ補助ツール`

目标不是自动交易，而是面向日本企业、金融顾问、投研团队提供：

- 日本市场默认配置
- 日文投資委員会式研究报告
- 风险与合规优先的研究辅助流程
- 可审计、可追溯的判断记录

产品化表达建议：

> 複数の専門エージェントが、株価・財務・ニュース・市場心理・リスクを分担分析し、投資委員会形式の日本語レポートを自動生成します。

## 2. 当前仓库现状

已具备的基础能力：

- 已支持东京证券交易所代码后缀 `.T`
- 默认 benchmark 映射中已包含 `.T -> ^N225`
- CLI 已支持 `Japanese` 作为输出语言选项
- 系统本身已有 analyst / researcher / risk / portfolio manager 多角色框架

当前与目标的主要差距：

- 默认模式仍偏全球通用，尚未切换为日本客户优先
- 输出结构仍偏通用交易分析，不是固定的投資委員会式报告
- 合规提示、研究边界、审计记录还不够显式
- 日本专属数据源仍未接入，只是后续扩展方向
- 本地化仍停留在“语言切换”，未深入到日期、货币、公司名、财务术语表达

## 3. 产品设计原则

- 默认先服务日本股票研究场景，而不是泛市场交易演示
- 默认输出研究建议，不直接执行交易
- 默认强调可解释性、审计留痕、反方观点、风险披露
- 默认使用更符合日本企业沟通习惯的措辞，而不是美式 `BUY/SELL signal`

## 4. 分阶段实施计划

### Phase 1: 日本市场默认模式

目标：让用户一进入系统，就感受到这是面向日本股票研究的版本。

实施内容：

- 将默认输出语言从 `English` 调整为 `Japanese`
- 增加日本市场模式配置，例如：
  - `market_profile = jp_equity`
  - `default_exchange_suffix = .T`
  - `default_benchmark = ^N225`
  - 可选支持 `^TOPX`
- 对未带交易所后缀的日本股票输入，设计补全策略
  - 例如用户输入 `7203` 时可提示或自动规范为 `7203.T`
- 本地化显示格式：
  - 日期使用日本客户熟悉的表达
  - 货币优先显示 `JPY / 円`
  - 公司名尽量保留日文或官方名称
  - 财务术语采用日文金融表达

交付结果：

- 日本模式配置项
- 默认 CLI / 配置行为调整
- 基础本地化规范文档

验收标准：

- 新用户在默认配置下可直接分析 `.T` 股票
- 报告默认输出日文
- benchmark 在日本股票场景下自动使用 `^N225` 或配置指定的 `^TOPX`

### Phase 2: 投資委員会式报告模板

目标：把最终输出从“通用分析结果”改造成日本客户更容易接受的正式研究报告。

强制输出字段：

- `投資判断`: `買い / 中立 / 売り`
- `確信度`
- `主要根拠`
- `反対意見`
- `最大リスク`
- `次に確認すべき材料`

实施内容：

- 重写 research manager / portfolio manager 的最终汇总模板
- 将 analyst 与 debator 输出整理为固定结构
- 为每个结论保留对应证据来源摘要
- 统一日文金融措辞，减少英文标签和美式投顾语气

交付结果：

- 标准化日文报告 schema
- 最终报告 prompt / formatter 改造
- 示例输出样本

验收标准：

- 每次输出都包含全部必填字段
- 即使模型自由生成，也会被结构化层强制约束回固定 schema
- 结果可直接给日本客户或内部投委会阅读

### Phase 3: 风险与合规优先

目标：明确系统边界，降低“自动投顾”或“自动交易”误解。

实施内容：

- 在 CLI、报告头部、导出结果中明确标注：
  - `投資助言ではありません`
- 禁止默认执行真实交易，仅生成研究建议
- 将“下单”相关流程降级为模拟 / 审批后动作
- 保留每次判断的关键元数据：
  - 输入 ticker
  - 分析日期
  - 模型与 provider
  - 使用的数据源
  - 最终判断
  - 主要理由
  - 生成时间

交付结果：

- 合规提示统一文案
- 审计日志结构
- 研究建议与执行动作的边界说明

验收标准：

- 所有主要入口都能看到非投顾声明
- 默认路径不直接触发真实交易动作
- 每次分析结果都能回溯输入、模型、时间与理由

### Phase 4: 日本客户专属数据层

目标：把“会说日文”的工具，升级成“真的懂日本市场”的工具。

优先数据方向：

- 日本経済新聞
- TDnet
- EDINET
- 有価証券報告書
- 日银政策
- 汇率 `USD/JPY`
- `JGB` 利率
- 东证行业分类
- `TOPIX / Nikkei` 对比
- 决算短信摘要

实施策略：

- 第一阶段先设计数据接口抽象，不急于一次性全接
- 先补“宏观 + 披露 + 财报摘要”三类最能提升可信度的数据
- 每个数据源增加来源标识与抓取时间，支持审计

交付结果：

- 日本数据源接入优先级列表
- 统一 data provider 接口
- 首批日本数据源适配方案

验收标准：

- 报告中可明确区分全球通用数据与日本本地数据
- 日本股票报告能引用至少一类日本专属信息源

### Phase 5: 产品化包装

目标：让该项目从开源框架表达，切换成可对日本客户说明价值的产品叙事。

建议产品名称：

- `日本株 AI リサーチ補助ツール`
- `日本株 投資委員会 Copilot`
- `JP Equity Research Copilot`

建议卖点：

- 多智能体分工分析日本股票
- 自动生成投資委員会形式日文报告
- 强调反方观点与最大风险，不只给单边结论
- 保留分析过程与判断依据，便于内部审计
- 先做研究辅助，再考虑执行联动

适合的客户画像：

- 日本中小型资管或精品投研团队
- 企业内部投资委员会
- 金融顾问与研究支持团队
- 面向日本市场的 fintech 服务商

## 5. 推荐实施顺序

推荐先做：

1. 默认日本模式
2. 投資委員会式报告 schema
3. 合规提示与审计日志
4. 日本数据源接口抽象
5. 首批日本专属数据接入
6. README / CLI / Demo 文案产品化

原因：

- Phase 1 到 Phase 3 能最快把项目从“通用研究框架”变成“可向日本客户演示的版本”
- Phase 4 决定长期壁垒，但不必阻塞前面的 demo 和包装

## 6. 技术改造建议

优先可能涉及的模块：

- `tradingagents/default_config.py`
  - 调整默认语言、market profile、benchmark 配置
- `cli/main.py` 与 `cli/utils.py`
  - 调整默认交互、报告语言默认值、合规提示
- 最终决策与汇总节点
  - 强制输出投資委員会 schema
- `tradingagents/agents/utils/structured.py`
  - 增加日本报告结构化约束
- 审计与结果记录相关模块
  - 扩充日志字段和导出格式
- 数据流模块 `tradingagents/dataflows/*`
  - 为日本本地数据源预留 provider 接口

## 7. MVP 定义

第一版可交付 MVP：

- 输入日本股票代码，如 `7203.T`
- 自动选择 `^N225` 或指定 `^TOPX` 作为 benchmark
- 默认输出日文报告
- 最终报告严格包含：
  - `投資判断`
  - `確信度`
  - `主要根拠`
  - `反対意見`
  - `最大リスク`
  - `次に確認すべき材料`
- 报告内明确标注 `投資助言ではありません`
- 保留每次分析的输入、时间、模型、理由

这版就已经足够用于：

- 日本客户 demo
- 内部 PoC
- 产品定位验证

## 8. 成功标准

如果以下条件成立，就说明日本客户化方向基本跑通：

- 用户默认体验明显偏向日本股票研究
- 输出结果读起来像日文投委会材料，而不是英文交易机器人翻译件
- 风险边界清晰，不会被误解为自动投顾或自动下单工具
- 分析结果具备基本审计追踪能力
- 后续可以平滑接入日本专属数据源继续增强壁垒

## 10. End-to-End 确认

从代码链路角度看，当前已具备完整 end-to-end 形状：

1. 用户输入日本股票代码，例如 `7203`
2. CLI 规范化为 `7203.T`
3. 系统按 `jp_equity` 默认模式运行
4. analysts / researchers / risk / portfolio manager 完成全流程分析
5. 最终输出固定的日文投委会报告
6. 导出报告头部显示实际运行 benchmark
7. JSONL audit log 保存完整判断字段与多行理由
8. 日本专属数据接口可在流程中被调用并保留 provenance

但需要明确：

- 我们目前确认的是“代码链路 end-to-end 完整”
- 不是“真实运行验证 end-to-end 完成”
- 因为本轮没有实际启动程序、也没有真实执行测试

因此最准确的结论是：

> 当前仓库已经具备 end-to-end 的框架与文档闭环；  
> 若要声明运行层面的 end-to-end 已验证，还需要补一次实际执行验证。
